<Notes>
You’re probably thinking “duh,” because component thinking is so commonplace
now. But it wasn’t back then. At least not in terms of front-end.

And to be clear, I’m NOT claiming CSS brought component thinking to
software—that goes back to IBM in the 60s. Software was well aware of
components. Also, JS has always borrowed object-oriented ideas since its
inception. One might claim object-oriented is a form of component.

But nothing in CSS’ spec suggests component thinking, as we just saw. And
connecting the dots to switch from the top-down approach (layout first, then
details & tiny elements later) to bottom-up approach (components first;
layouts are just arrangements of components), this approach led down the
current path we’ve followed for almost 10 years.

</Notes>

# Alternates / State

<SyntaxHighlighter
  language="jsx"
  style={monokaiSublime}
>{`const Button = styled.button\`
  background-color: \${props => props ? '#006dcc' : '#f00f0f0'};
  color: \${props => props.cta ? '#fff' : '#333'};
  opacity: \${props => props.disabled ? 0.6 : 1};
  pointer-events: \${props => props.disabled ? 'not-allowed' : default};
\`;

<Button>Log in</Button>
<Button cta disabled>Sign up</Button>`}</SyntaxHighlighter>

<Notes>
Yes, I guess I did just recreate Twitter Bootstrap in Styled Components 🤷‍♂️

Can you see how far we’ve come in a few short years?

Also, this is such a huge leap forward because we can actually type-check,
and pass complex values to our CSS in ways we couldn’t before.

Also we didn’t talk about it, but encapsulation is solved. No specificity
fighting! No name memorization! This is expressive and clear.

</Notes>
